Context

A fraud module where users couldn't find what they needed — and no one agreed on what it should do.

FYNDNA's CIC module sits at the intersection of fraud detection and document risk assessment. It serves users who need to spot fraudulent documents quickly — but the module had consistently high attrition. Users dropped off before completing their task. At the same time, multiple internal stakeholder teams had overlapping and sometimes contradicting requirements for how the module should behave. Shrea was brought in as a consultant through Lollypop Design to untangle this — and design within an existing design library she didn't author.

Challenge 01

The problem statement was a moving target

CIC sits inside a product that spans fraud detection, document verification, and risk management. Each team described the problem differently. Before a single screen could be designed, the problem itself had to be defined — and agreed on.

The module served users from multiple internal teams — each with a different idea of what "completing a task" meant. Designing for one broke the flow for another.

What made this hard

  • No single owner — requirements came from multiple teams simultaneously
  • Each team described the same screens in different terms
  • Ambiguity grew as the problem was explored, not shrunk

How I approached it

  • Mapped all team requirements to identify overlaps and true contradictions
  • Used UX best practices to propose a single shared definition of the task
  • Held alignment sessions before touching the design library

What this changed in how I design

Challenge 02

Multiple stakeholders, contradicting requirements

Once the problem was defined, the harder challenge emerged: each team had legitimate needs that directly conflicted with each other. A decision that satisfied Team A created friction for Team B. There was no neutral solution — only trade-offs that needed to be argued and agreed.

"Every team thought the other team's requirement was optional. None of them were."
The contradiction wasn't a communication failure — the teams genuinely needed different things from the same screen. The design had to surface that trade-off explicitly and help stakeholders decide, not just compromise.

What made this hard

  • No single stakeholder had final authority
  • Each team interpreted "user needs" through their own operational lens
  • Compromises that split the difference often satisfied no one

How I navigated it

  • Presented trade-offs explicitly — not solutions, but decisions for stakeholders to make
  • Used UX principles as a neutral framework to evaluate each option
  • Built consensus by showing how each direction affected real user tasks

Challenge 03

High attrition on a module users couldn't navigate

Users were dropping off the CIC module before completing their task. The attrition wasn't caused by a single broken flow — it was systemic. Users couldn't find what they needed quickly enough, and the cost of confusion in a fraud detection context is high.

In fraud and risk tooling, speed of task completion matters more than comprehensive feature sets. A user who can't find the right document check quickly will abandon the tool and find a workaround.

What made this hard

  • The module was feature-rich but information-dense — clarity required subtraction
  • Every team wanted their feature visible — reducing noise was politically difficult
  • Designing inside an existing library constrained the options available

Design directions explored

  • Prioritised task entry points over feature breadth on primary screens
  • Used the design library's existing hierarchy to create clearer visual weight
  • Reduced the number of decisions a user had to make before reaching their task
